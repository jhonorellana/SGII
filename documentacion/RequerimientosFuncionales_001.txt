1. Introducción
1.1 Propósito
El presente documento describe los requerimientos funcionales del Sistema de Gestión Integral de Inversiones (SGII), cuyo objetivo es permitir la administración, control, análisis y valorización de inversiones financieras pertenecientes a uno o varios grupos familiares o clientes.
El sistema permitirá gestionar instrumentos financieros tales como:
* Bonos 
* Obligaciones 
* Papeles comerciales 
* Titularizaciones 
* Facturas comerciales 
* Acciones 
* Valores genéricos 
* Otras inversiones 
* Saldos externos (cuentas, deudas, otros valores) 
Además, permitirá:
* administrar tablas de amortización 
* analizar transacciones históricas del mercado 
* cargar vectores diarios de precios 
* generar reportes financieros 
* administrar múltiples usuarios y grupos familiares 
* migrar datos desde el sistema actual 

2. Alcance del sistema
El sistema permitirá:
* Gestión operativa
* registrar inversiones 
* registrar ventas parciales 
* registrar ventas totales 
* calcular tablas de amortización 
* actualizar estados financieros 
* Gestión analítica
* visualizar flujos futuros 
* analizar vencimientos 
* consolidar capital 
* analizar transacciones del mercado 
* analizar precios históricos 
* valorar inversiones con precios del mercado 
* Gestión administrativa
* administrar usuarios 
* administrar grupos familiares 
* manejar catálogos parametrizables 
* gestionar migración de datos 

3. Objetivos del sistema
El sistema deberá permitir:
1. Gestionar el portafolio de inversiones vigente. 
2. Analizar el comportamiento del mercado financiero. 
3. Valorar diariamente las inversiones. 
4. Facilitar la toma de decisiones financieras. 
5. Automatizar cálculos financieros. 
6. Mantener trazabilidad histórica. 
7. Escalar a múltiples clientes o familias.

6.1 Gestión de Catálogos (Reference – Dictionary)
* RF-001: El sistema debe permitir crear catálogos utilizando la estructura reference y dictionary. 
* RF-002: El sistema debe permitir crear, editar, activar e inactivar valores dentro de cada catálogo. 
* RF-003: Todos los estados del sistema deben manejarse mediante catálogos. 
* RF-004: El sistema debe permitir administrar catálogos como: 
o Tipos de inversión 
o Estados de inversión 
o Estados de amortización 
o Tipos de usuario 
o Tipos de propietario 
o Tipos de aportante 
o Tipos de moneda 
o Tipos de instrumentos 
o Tipos de cuenta 
o Tipos de periodicidad 
* RF-005: El sistema debe permitir consultar los valores activos de cada catálogo. 
* RF-006: El sistema debe permitir mantener históricos de cambios en catálogos. 

6.2 Gestión de Usuarios
* RF-007: El sistema debe permitir registrar usuarios. 
* RF-008: El sistema debe permitir asignar roles a usuarios.

7. Requerimientos No Funcionales
7.1 Seguridad
* RNF-001: El sistema debe cifrar contraseńas. 
* RNF-002: El sistema debe implementar control de acceso por roles. 
* RNF-003: El sistema debe registrar auditoría de cambios. 
* RNF-004: El sistema debe registrar historial de modificaciones. 

7.2 Rendimiento
* RNF-005: El sistema debe responder consultas en menos de 3 segundos. 
* RNF-006: El sistema debe soportar ejecución de procesos diarios automatizados. 
* RNF-007: El sistema debe permitir manejar grandes volúmenes históricos. 

7.3 Integridad de datos
* RNF-008: El sistema debe utilizar integridad referencial. 
* RNF-009: El sistema debe usar motor de base de datos transaccional (InnoDB). 
* RNF-010: El sistema debe evitar duplicación de datos. 

7.4 Disponibilidad
* RNF-011: El sistema debe permitir respaldos automáticos. 
* RNF-012: El sistema debe permitir restauración de respaldos. 

7.5 Escalabilidad
* RNF-013: El sistema debe permitir múltiples grupos familiares. 
* RNF-014: El sistema debe soportar crecimiento en volumen de datos. 
* RNF-015: El sistema debe permitir integración futura con fuentes externas.
