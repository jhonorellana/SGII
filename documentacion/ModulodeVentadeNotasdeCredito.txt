Módulo de Venta de Notas de Crédito – Especificación Funcional

Contexto del negocio
En Ecuador las Notas de Crédito normalmente representan:
* Notas de crédito del SRI 
* Certificados tributarios 
* Instrumentos utilizados para compensar impuestos u obligaciones 
El modelo de negocio utilizado en este sistema se basa principalmente en:
* Compra de notas de crédito con descuento 
* Acumulación de múltiples notas pequeńas 
* Venta consolidada de varias notas a un porcentaje mayor 
Ejemplo:
Compra:
Nota 1 -> 95%
Nota 2 -> 94%
Nota 3 -> 96%

Venta consolidada:
97%
La utilidad se genera por la diferencia entre:
Valor de compra consolidado
vs
Valor de venta consolidado

Modelo esperado de funcionamiento
Escenario principal
El usuario puede:
1. Comprar múltiples notas de crédito pequeńas 
2. Almacenarlas individualmente como inversiones independientes 
3. Seleccionar varias notas para realizar una sola venta agrupada 
4. Registrar un único ingreso real de dinero en la cuenta bancaria 
5. Obtener estadísticas individuales de rentabilidad por cada nota 

Consideraciones importantes
Una venta puede involucrar múltiples inversiones
La funcionalidad NO debe asumir:
1 venta = 1 inversión
Debe soportar:
1 venta = N inversiones
Porque una sola operación de venta puede agrupar:
* varias notas de crédito 
* distintos montos nominales 
* distintas fechas de compra 
* distintos porcentajes de compra 
Diseńo esperado
* Tabla principal: venta_inversion
Debe representar:
* la venta consolidada/global 
* el valor total recibido 
* el porcentaje de venta aplicado 
* las comisiones 
* el ingreso bancario real 
Esta tabla representa:
La operación comercial real

Nueva tabla requerida: venta_inversion_detalle
Debe crearse una nueva tabla intermedia para asociar múltiples inversiones a una venta.
Objetivo
Permitir:
* asociar varias notas de crédito a una venta 
* calcular utilidad individual 
* mantener trazabilidad histórica 
* generar estadísticas por inversión 

Estructura esperada de venta_inversion_detalle
CREATE TABLE venta_inversion_detalle (
  id_venta_inversion_detalle INT NOT NULL AUTO_INCREMENT,

  id_venta_inversion INT NOT NULL,
  id_inversion INT NOT NULL,

  valor_nominal DECIMAL(18,2) NOT NULL,
  valor_compra DECIMAL(18,2) NOT NULL,

  porcentaje_compra DECIMAL(18,8) DEFAULT NULL,

  valor_venta_asignado DECIMAL(18,2) NOT NULL,

  porcentaje_venta DECIMAL(18,8) DEFAULT NULL,

  utilidad DECIMAL(18,2) DEFAULT 0.00,

  rendimiento DECIMAL(18,8) DEFAULT 0.00000000,

  fecha_creacion DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,

  fecha_actualizacion DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP
  ON UPDATE CURRENT_TIMESTAMP,

  PRIMARY KEY (id_venta_inversion_detalle),

  KEY idx_venta_detalle_venta (id_venta_inversion),

  KEY idx_venta_detalle_inversion (id_inversion),

  CONSTRAINT fk_venta_detalle_venta
    FOREIGN KEY (id_venta_inversion)
    REFERENCES venta_inversion(id_venta_inversion),

  CONSTRAINT fk_venta_detalle_inversion
    FOREIGN KEY (id_inversion)
    REFERENCES inversion(id_inversion)
);

Flujo esperado de la funcionalidad
Paso 1 – Selección de inversiones
El usuario debe poder:
* visualizar notas de crédito activas 
* seleccionar múltiples inversiones 
* ver: 
o valor nominal 
o porcentaje de compra 
o valor invertido 
o utilidad esperada 

Paso 2 – Venta consolidada
El usuario ingresará:
* porcentaje de venta
o 
* valor total recibido 
El sistema deberá:
* calcular automáticamente: 
o valor consolidado 
o utilidad total 
o utilidad individual por nota 
o ROI individual 

Paso 3 – Creación de venta principal
El sistema debe crear:
1 registro en venta_inversion
Representando:
* la venta global 
* el ingreso total real 
* las comisiones 
* el valor final recibido 

Paso 4 – Creación de detalles
El sistema debe crear:
N registros en venta_inversion_detalle
Uno por cada inversión seleccionada.
Cada detalle debe almacenar:
* monto asignado de la venta 
* utilidad individual 
* porcentaje de venta aplicado 
* rendimiento individual 

Paso 5 – Actualización de inversiones
Las inversiones involucradas deben:
* cambiar su estado 
* marcarse como vendidas 
* registrar fecha de venta 
Movimiento de capital esperado
Importante
Debe registrarse:
UN SOLO movimiento de capital
Porque en la realidad:
* el banco recibe una sola transferencia 
* no múltiples ingresos separados 

Tabla movimiento_capital
Debe crearse:
1 registro
Con:
* tipo movimiento = venta inversión 
* monto = valor total recibido 
* id_venta_inversion = venta principal 
* id_inversion = NULL 

Regla importante
movimiento_capital.id_inversion
Debe permitir:
NULL
Porque el movimiento representa:
una venta agrupada
No una inversión individual.

Distribución de utilidad
La utilidad debe calcularse individualmente.
Ejemplo:
NotaCompraVenta asignadaUtilidadNota 195097020Nota 21900194040Nota 32850291060
Beneficios esperados del diseńo
Este enfoque permitirá:
* conciliación bancaria correcta 
* trazabilidad real de operaciones 
* estadísticas individuales 
* análisis histórico por nota 
* cálculo de ROI individual 
* soporte para ventas agrupadas 
* soporte para futuras funcionalidades tributarias 

Consideraciones futuras
Este diseńo permitirá posteriormente:
* ventas parciales 
* recompras 
* reutilización tributaria 
* agrupación por empresa/emisor 
* métricas avanzadas de rendimiento 
* dashboards de arbitraje 
* estadísticas por porcentaje de compra/venta 

Resumen conceptual
* Lo importante es separar:
Venta comercial real
venta_inversion
De:
Distribución financiera interna
venta_inversion_detalle
Y mantener:
1 solo movimiento bancario real
