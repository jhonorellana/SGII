Vamos a diseńar el Modelo Conceptual de Datos procurando:
* mantener una estructura cercana a la actual para facilitar la migración 
* usar nombres en espańol 
* separar claramente: 
o tablas núcleo 
o tablas derivadas 
o tablas históricas de mercado 
o catálogos 
* dejar preparado el modelo para crecer sin romper demasiado lo que ya tienes 
Voy a plantearlo como modelo conceptual propuesto, no todavía como SQL físico.

1. Criterios de diseńo
1.1 Mantener cercanía con el sistema actual
Se conservará la lógica principal actual basada en:
* inversión 
* amortización 
* otros valores 
* balance / consolidado 
* snapshots diarios 
* históricos de mercado 
1.2 Rediseńar solo donde realmente hace falta
Se recomienda conservar la esencia funcional, pero corrigiendo estos puntos:
* propietario ya no como texto libre 
* aportante separado de propietario 
* estados como catálogos 
* mejor normalización de personas y grupos familiares 
* tablas derivadas claramente separadas de tablas transaccionales 
1.3 Nombres en espańol
Las entidades principales se definirán en espańol, por ejemplo:
* inversion 
* amortizacion 
* grupo_familiar 
* persona 
* usuario 
* emisor 
* instrumento 
* catalogo 
* catalogo_valor

9. Recomendaciones finales del modelo conceptual
9.1 Mantener inversion como núcleo
Eso es lo mejor para no romper el sistema ni complicar la migración.
9.2 No meter demasiada lógica en tablas derivadas
variation, consolidado_inversiones y acciones_ultimas_transacciones deben seguir viéndose como resultados de proceso diario.
9.3 Usar catálogos para todo estado o tipo
Eso te dará mucha flexibilidad.
9.4 Separar persona de usuario
No toda persona necesita login, pero toda inversión sí necesita relacionarse con personas.
9.5 Separar instrumento de inversión
Eso te permitirá crecer mejor cuando quieras enriquecer análisis de mercado.

7.8 Relaciones lógicas principales
* rol 1 — N usuario 
* persona 1 — 0..1 usuario 
* grupo_familiar 1 — N grupo_familiar_persona 
* persona 1 — N grupo_familiar_persona 
* catalogo 1 — N catalogo_valor 
* emisor 1 — N instrumento 
* grupo_familiar 1 — N inversion 
* persona 1 — N inversion como propietario 
* persona 1 — N inversion como aportante 
* instrumento 1 — N inversion 
* emisor 1 — N inversion 
* inversion 1 — N amortizacion 
* inversion 1 — N venta_inversion 
* inversion 1 — N documento_inversion 
* grupo_familiar 1 — N otro_valor 
* grupo_familiar 1 — N consolidado_inversion 
* grupo_familiar 1 — N variacion_capital 
* variacion_capital 1 — N variacion_capital_detalle 
* persona 1 — N variacion_capital_detalle 

7.9 Reglas lógicas importantes
7.9.1 Estados por catálogo
Todos los estados y tipos deben referenciar catalogo_valor.
7.9.2 Propietario y aportante
Deben ser FK distintas a persona.
7.9.3 Tablas derivadas
consolidado_inversion, variacion_capital, variacion_capital_detalle y acciones_ultimas_transacciones no son transaccionales; se regeneran por proceso diario.
7.9.4 Históricos de mercado
Deben mantenerse compatibles con la fuente externa y no sobre-normalizarse.

7.10 Mapeo lógico desde el sistema actual
* Migración casi directa
* inversion ? inversion 
* amortizacion ? amortizacion 
* othervalue ? otro_valor 
* reference ? catalogo 
* dictionary ? catalogo_valor 
* Migración con transformación
* balance ? otro_valor 
* variation ? variacion_capital + variacion_capital_detalle 
* consolidado_inversiones ? consolidado_inversion 
* shares_lastdate ? acciones_ultimas_transacciones 
* Migración compatible
* históricos de mercado ? tablas *_his equivalentes en espańol 

7.11 Recomendaciones de tipos de datos
* Para identificadores
* INT o BIGINT según crecimiento esperado

* DATE para fechas puras
